# Repository Guidelines

This is an out-of-the-box go-kratos service template. This file owns the
layering contract agents must follow when changing the template.

## Project Snapshot

The template demonstrates a protobuf-first Go backend with HTTP and gRPC
transports, Wire dependency injection, OpenAPI generation, config generation,
and a small in-memory Todo CRUD example.

- Current Go module: `helloworld`.
- Current app entrypoint: `cmd/helloworld`.
- Current sample API: `api/todo/v1`.
- There is no frontend app in this repository.
- The data layer wires generic SQL and Redis clients from config. The sample
  Todo repository is intentionally in-memory.
- JWT, Casbin, zaplog, and same-name type copying live under `internal/pkg`.

## Project Skills

Use project-specific skills in `.agents/skills/` when the task matches:

- `kratos-backend-api`: proto/API/service/biz/data endpoint work, route
  registration, error contracts, or Wire provider wiring.
- `kratos-ent-data`: optional Ent-backed persistence work, Ent schema changes,
  repositories, transactions, Redis-backed state, or persistence mapping.
- `kratos-devops-quality`: local startup, Makefile targets, generation,
  config, OpenAPI, tests, Docker, CI, or deployment troubleshooting.

Before non-trivial work, read this file and the relevant skill. Use `rg` for
exact identifiers, files, generated symbols, and import paths.

## Commands

```bash
make init       # Install local codegen tools used by this template
make api        # Regenerate API proto files and openapi.yaml
make config     # Regenerate internal config proto
make generate   # Run go generate and go mod tidy
make all        # api + config + generate
make build      # Compile ./cmd/helloworld to ./bin/helloworld
make run        # Run the service with ./configs
make lint       # Run golangci-lint
make test       # Run go test ./...

# Direct run
go run ./cmd/helloworld -conf ./configs
```

Run `make all` after proto, config proto, or Wire provider changes. Run
`make lint` after Go code changes. Run the smallest relevant `go test` package
for code changes; use `make test` when shared behavior changed.

## Project Structure

```text
api/<domain>/<version>/   Proto sources and generated stubs. Public contract.
cmd/<app>/                Entrypoint, Wire injector, main.go.
configs/                  Runtime config (config.yaml). No secrets.
internal/conf/            Config proto; generated by make config.
internal/server/          HTTP/gRPC server wiring.
internal/service/         Transport adapters; one file per resource.
internal/biz/             Domain models, usecases, repo interfaces, errors.
internal/data/            Repo implementations and storage clients.
internal/pkg/             Shared auth, token, and conversion helpers.
.agents/skills/           Project agent skills.
openapi.yaml              Generated OpenAPI document.
```

## Layering and Dependency Rules

Three model shapes flow through three layers. `biz` owns the DO, `data` owns
the PO, and `service` converts at the transport boundary.

```text
client -> DTO -> service -> DO -> biz -> DO -> data -> PO -> storage
                              ^              ^
                              | declares     | implements
                              +-- repo IF ---+

DTO  Data Transfer Object: proto request / response.
DO   Domain Object: pure biz model, no proto, no storage tags.
PO   Persistent Object: storage shape, owned by data.
```

| Layer   | Owns | Speaks at boundary | Never speaks            |
|---------|------|--------------------|-------------------------|
| service | -    | DTO <-> DO         | PO, storage client      |
| biz     | DO   | DO                 | DTO, PO, storage client |
| data    | PO   | DO <-> PO          | DTO                     |

- `service` imports `api/...` for DTOs and `biz` for DOs. Never import `data`.
- `biz` imports `api/...` only for error reason enums. Never import `service`
  or `data`. Repo interfaces are declared here.
- `data` imports `biz` to implement repo interfaces. Never import `service`
  or DTOs.
- `cmd` is the only place that wires all layers through Wire.

A change crossing these arrows the wrong way is a layering bug; fix the design
instead of adding the import.

## Layer Responsibilities

### service: DTO <-> DO

- Embed `Unimplemented<Resource>ServiceServer`.
- Parse and validate request inputs at the service boundary.
- Convert incoming proto messages into DOs with local `convert<Resource>`
  helpers.
- Build proto responses inline at return sites or with small local helpers.
- Parse AIP list requests via `filtering`, `ordering`, and `pagination`.
- Apply `fieldmask.Update` for partial updates.
- Return `biz` errors. No business rules, no storage access, no PO.

### biz: DO only

- Own the DO (`type <Resource> struct`) with no proto or storage tags.
- Own the usecase and repo interface (`type <Resource>Repo interface`).
- Own typed errors built with Kratos errors plus API error reason enums.
- Own `ListOption` helpers such as `ListFilter`, `ListOrderBy`, `ListOffset`,
  and `ListLimit`, so callers do not leak storage primitives.
- Put validation, orchestration, idempotency, authorization calls, and
  transaction boundaries here when the service grows.

### data: DO <-> PO

- Implement `biz.<Resource>Repo`.
- Constructors return the interface, not the concrete repo:
  `func New<Resource>Repo(d *Data) biz.<Resource>Repo`.
- Define a PO only when storage shape diverges from the DO. PO types stay in
  `internal/data`.
- Use free functions such as `new<Resource>` (DO to PO) and `toBiz` (PO to DO)
  when conversion is needed.
- Long-lived storage clients live in `*Data`; repos receive `*Data` and do not
  construct their own clients.
- Translate storage errors to `biz` typed errors so upper layers never branch
  on driver-specific errors.

### server

- Construct HTTP/gRPC servers, apply middleware, and register services.
- No translation, business logic, persistence, or storage client access.
- Keep JWT and Casbin middleware wiring here. Match whitelist entries by
  generated operation constants, not URL strings.

### internal/pkg

- `internal/pkg/token` owns JWT generation and validation.
- `internal/pkg/authz` owns current-user context helpers and Casbin middleware.
- `internal/pkg/zaplog` owns zap-backed structured logging, file rotation, and
  log masking helpers.
- `internal/pkg/typecatch` owns small same-name struct copy helpers. Prefer
  explicit converters for renamed fields or cross-field semantics.

## Add-a-Resource Checklist

1. DTO: define `Create<Resource>`, `Get<Resource>`, `List<Resources>`,
   `Update<Resource>`, and `Delete<Resource>` in `api/<domain>/<version>/`;
   then run `make api`.
2. DO and repo interface: declare both in `internal/biz`; build the usecase on
   top of the repo interface.
3. Repo implementation: implement in `internal/data`, returning
   `biz.<Resource>Repo`; add a PO and conversion helpers only if storage shape
   differs from the DO.
4. Wiring: register the repo constructor in `data.ProviderSet`, the usecase in
   `biz.ProviderSet`, the service in `service.ProviderSet`, and HTTP/gRPC
   services in `internal/server`.
5. Regenerate and verify: run `make all` and targeted tests.

## Template Customization Checklist

When creating a new service from this template:

1. Update the module path: `go mod edit -module github.com/your-org/your-app`.
2. Replace imports from `helloworld/...` with the new module path.
3. Rename `cmd/helloworld` if needed and update `APP` in `Makefile`.
4. Update proto `go_package` options and regenerate with `make all`.
5. Replace the sample Todo resource with your real domain model.
6. Update `configs/config.yaml`; never put real credentials in the template.
7. Set `auth.private_key_path` to a real PKCS8 Ed25519 private key path before
   production use; the empty default creates an ephemeral key for local startup.
8. Build, lint, run, and test with `make build`, `make lint`, `make run`, and
   `make test`.

## Testing Seam

Tests live beside the code they cover (`*_test.go`). Test layers in isolation:
service tests fake or compose a usecase, biz tests fake the repo, and data tests
exercise repo implementations at the storage boundary.

Every top-level `Test...` function and every `t.Run` subcase must have a
nearby Chinese `//` comment explaining the scenario. The
`internal/pkg/testcommentcheck` guard enforces this through `make test`.

For non-trivial logic, leave the smallest runnable check that would fail if the
logic regresses.

## Generated Files

Do not hand-edit generated code or generated docs:

- `api/**/*.pb.go`
- `api/**/*_grpc.pb.go`
- `api/**/*_http.pb.go`
- `internal/conf/conf.pb.go`
- `cmd/*/wire_gen.go`
- `openapi.yaml`

Regenerate from source files with `make api`, `make config`, or `make all`.

## Naming and Error Reasons

- Resource: `<Resource>` (for example, `Todo`).
- Collection RPC: `List<Resources>`.
- Types: `<Resource>Repo`, `<Resource>Usecase`, and `<Resource>Service`.
- PO types live inside `internal/data`; pick storage-shaped names and convert
  with `new<Resource>(do)` / `toBiz(po)` when needed.
- Error reasons are declared in `api/<domain>/<version>/error_reason.proto`
  and surfaced as `Err<Resource><Cause>` in `biz`.

## Security and Commits

- Conventional commit prefixes: `feat:`, `fix:`, `refactor:`,
  `chore(deps):`, `docs:`, and `test:`.
- Regenerated files belong in the same commit as their source changes.
- Never commit real credentials in `configs/config.yaml`.
- Do not print, copy, or summarize local secret values in responses, logs,
  docs, tests, or examples.
- Do not commit JWT private keys. Use config paths or deployment secrets.
- Mask credentials before they reach logs; use `zaplog.MaskPassword` or
  structured secret field names such as `password`.

## Final Verification Checklist

- Read the relevant skill under `.agents/skills/`.
- Run generation commands when proto, config proto, or Wire providers changed.
- Run `make lint` after Go code changes.
- Run targeted tests; broaden to `make test` for shared behavior.
- Inspect changed files and ensure generated files changed only as expected.
